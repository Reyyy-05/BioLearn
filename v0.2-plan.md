# 🛠️ BioLearn v0.2 — Development Plan ("Evaluasi Lebih Tajam")

> Derived from `brainstorming.md` §4. Constraints honored: **1 developer · 1–2 weeks · no backend · no new complex deps · Expo SDK 54 · existing stack (Expo RN, TS, Expo Router, Zustand, local seed data)**.
>
> Stack reality check (already installed, no new deps needed): `expo-haptics`, `react-native-reanimated` + `react-native-worklets`, `@expo/vector-icons`. v0.2 uses only these.

---

## 1. Final v0.2 Scope

Five shippable increments, each independently demo-able:

| # | Feature | Why it makes the cut |
|---|---|---|
| **A** | **Bigger question bank + per-attempt shuffle** (questions & options) | Highest impact / lowest cost; makes repeat-demos feel real. Pure data + logic. |
| **B** | **Quiz modes: Latihan vs Ujian + optional timer** | Core product differentiator (evaluation). Local state + one route param. |
| **C** | **Attempt history + best score per module** | Enables trend/“nilai tertinggi”; small store/type change. |
| **D** | **Exam-readiness score + smarter remedial ordering** | Sharpens the adaptive-recommendation story. Pure derived logic. |
| **E** | **Demo polish: haptics + animated progress ring + page transitions** | Premium feel for client demo; uses already-installed libs. |

**Out of v0.2 by design:** anything needing a backend, persistence, new dependencies, or a new role surface (see §2).

---

## 2. Features Rejected From v0.2 (and why)

| Rejected idea (from brainstorming) | Reason |
|---|---|
| **AsyncStorage persistence (Roadmap v0.3)** | Adds a dependency + breaks the "ephemeral, resets clean every demo" contract. Deferred to v0.3 as planned. |
| **Dashboard Guru (full)** | New screens + mock multi-student data model = too large for 1–2 weeks alongside A–E. Stays dummy. |
| **Admin Konten CRUD** | In-memory CRUD is deceptively large (forms, validation, edit/delete flows) and low demo value. Park. |
| **Backend / multi-user (v1.0)** | Explicitly out of scope by constraint. |
| **AI Tutor / adaptive ML (v2.0)** | Out of scope; no infra. |
| **New question types (true/false, matching)** | Each needs its own UI + grading path; bank expansion (A) gives more value per hour. |
| **Streak + badges** | Needs date math that fights the "reset demo" model; cosmetic. Revisit with v0.3 persistence. |
| **Full dark mode pass / i18n / a11y overhaul** | Theme foundation already exists; a dedicated pass is its own milestone. Polish (E) only touches new surfaces. |
| **Notes / bookmarks / flashcards / glossary** | Net-new content surfaces; nice-to-have, not aligned with v0.2's "sharper evaluation" theme. |

## 3. Checkpoint-Based Implementation Plan

Six checkpoints (CP0 baseline + A–E). Each ends green on `npx tsc --noEmit`, demo-flow intact, and `resetDemo()` clearing all new state.

---

### CP0 — Baseline guardrail (½ day)
**Goal:** Lock a known-good starting point and a quick verification ritual before touching code.

- **Files changed:** none (optionally a short `CONTRIBUTING`/notes line; skip if undesired).
- **New types/state:** none.
- **Acceptance criteria:**
  - `npx tsc --noEmit` clean.
  - `npx expo install --check` reports deps in sync.
  - Full demo flow in README runs on Expo Go without error.
- **Manual test steps:** Launch `npx expo start -c`; walk Login → Dashboard → Modules → Detail → Quiz → Result → Progress → Reset.
- **Git commit:** `chore: snapshot pre-v0.2 baseline, verify tsc + expo deps`

---

### CP-A — Expanded bank + per-attempt shuffle
**Goal:** Grow seed questions and randomize question order + option order per attempt, without breaking grading.

- **Files likely changed:**
  - `data/seedQuestions.ts` (3 → 6–8 questions per module).
  - `store/useLearningStore.ts` (`startQuiz` builds a shuffled per-attempt question list with remapped correct index).
  - `app/quiz.tsx` (read the shuffled list from store instead of filtering raw `questions`).
  - `types/index.ts` (new `ActiveQuizQuestion` shape; quiz config field).
- **New types/state:**
  ```ts
  // types
  interface ActiveQuizQuestion {
    id: string;
    moduleId: string;
    question: string;
    options: string[];          // already shuffled
    correctAnswerIndex: number; // remapped to shuffled options
    explanation: string;
  }
  // state
  activeQuiz: { moduleId: string; questions: ActiveQuizQuestion[] } | null;
  ```
  - Shuffle helper: Fisher–Yates. **No `Math.random` ban here** (app runtime, not a Workflow script) — but seed it per-call; keep it pure/local.
- **Acceptance criteria:**
  - Each module exposes ≥6 questions; quiz pulls a fresh random order each start.
  - Option order varies between attempts; correct answer still grades correctly (remapped index).
  - Score math unchanged (`correct/total*100`, KKM 70).
  - `resetDemo` clears `activeQuiz`.
- **Manual test steps:** Start a quiz twice on the same module → order/options differ. Deliberately pick the known-correct answer → feedback shows "Benar"; pick wrong → "Salah". Finish → result score matches manual count.
- **Git commit:** `feat(quiz): expand question bank and shuffle questions + options per attempt`

---

### CP-B — Quiz modes (Latihan / Ujian) + optional timer
**Goal:** Two modes. **Latihan** = instant per-question feedback (current behavior). **Ujian** = no feedback until the end + optional countdown timer; auto-submit on timeout.

- **Files likely changed:**
  - `app/module-detail.tsx` (mode picker before "Mulai Kuis"; pass `mode` param).
  - `app/quiz.tsx` (branch feedback behavior on mode; add timer UI + auto-submit).
  - `app/quiz-result.tsx` (show mode + time used).
  - `store/useLearningStore.ts` (store `mode` on `activeQuiz`; `finishQuiz` records mode/duration).
  - `types/index.ts` (`QuizMode`, timer fields).
- **New types/state:**
  ```ts
  type QuizMode = 'practice' | 'exam';
  // activeQuiz gains: mode: QuizMode; startedAt: string; durationSec: number | null;
  // QuizResult gains: mode: QuizMode; elapsedSec: number;
  ```
- **Acceptance criteria:**
  - Latihan behaves exactly as today (feedback after each "Cek Jawaban").
  - Ujian hides explanations until submission; shows running timer; at 0s it auto-finishes and routes to result.
  - Timer interval is cleaned up on unmount / early exit (no leaked intervals, no setState-after-unmount warning).
  - Mode shown on result screen.
- **Manual test steps:** Detail → choose **Ujian** with timer → confirm no feedback per question, timer counts down, let it hit 0 → lands on result. Repeat with **Latihan** → feedback per question, no timer. Exit mid-exam via "Keluar Kuis" → no timer-keeps-running warning.
- **Git commit:** `feat(quiz): add practice vs exam modes with optional countdown timer`

---

### CP-C — Attempt history + best score
**Goal:** Persist (in-memory) every attempt and surface "nilai tertinggi" + a small attempt trend.

- **Files likely changed:**
  - `store/useLearningStore.ts` (`finishQuiz` appends to history; track `bestScore`).
  - `types/index.ts` (`QuizAttempt`; extend `LearningProgress`).
  - `app/quiz-result.tsx` (show best score + "naik/turun vs sebelumnya").
  - `app/progress.tsx` (best score per module; attempt count already shown).
- **New types/state:**
  ```ts
  interface QuizAttempt {
    moduleId: string; mode: QuizMode;
    score: number; correctCount: number; totalQuestions: number;
    passed: boolean; completedAt: string; elapsedSec: number;
  }
  // LearningProgress gains: bestScore: number | null;
  // state gains: attemptHistory: QuizAttempt[];
  ```
- **Acceptance criteria:**
  - Each finished quiz appends one `QuizAttempt`.
  - `bestScore` is the max across attempts; never decreases when a later attempt is lower.
  - Result screen shows current vs best and delta vs previous attempt.
  - `resetDemo` clears `attemptHistory` and `bestScore`.
- **Manual test steps:** Take a module quiz scoring low, then again scoring high → best reflects high, history length = 2. Take it a third time lower → best unchanged, delta shows "turun". Reset → history empty.
- **Git commit:** `feat(progress): record quiz attempt history and best score per module`

---

### CP-D — Exam-readiness score + smarter remedial ordering
**Goal:** A single 0–100 "Kesiapan Ujian" indicator per grade and globally, and remedial list ordered by distance below KKM.

- **Files likely changed:**
  - `store/useLearningStore.ts` *or* a new pure `utils/readiness.ts` (derive readiness; **no new dep**, plain TS).
  - `app/progress.tsx` (readiness card; sort `lowScoreModules` by `70 - bestScore` descending).
  - `app/dashboard.tsx` (compact readiness ring/number + "lanjutkan dari terakhir").
- **New types/state:**
  - No new persisted state — **derived selectors only** (keeps store lean and reset-safe).
  - Optional: `getReadiness(state): { overall: number; byGrade: Record<number, number> }`.
  - Formula (transparent, demo-explainable): `readiness = round(0.5 * videoCompletionRate + 0.5 * avgBestScoreNormalized)`.
- **Acceptance criteria:**
  - Readiness = 0 on fresh/reset; rises as videos watched + scores improve; 100 only when all videos watched and all best scores = 100.
  - Remedial list ordered worst-first by gap to 70.
  - Dashboard shows readiness without extra store fields (pure derive).
- **Manual test steps:** From reset, readiness = 0. Watch one video → rises. Pass one quiz → rises further. Two modules below 70 with scores 40 and 65 → 40 listed first.
- **Git commit:** `feat(progress): add exam-readiness score and gap-based remedial ordering`

---

### CP-E — Demo polish (haptics + animated ring + transitions)
**Goal:** Premium tactile/visual feel using only installed libs.

- **Files likely changed:**
  - `app/quiz.tsx` (haptic on correct/wrong/submit via `expo-haptics`, web-guarded).
  - `components/ProgressRing.tsx` (new; reanimated animated ring) — reused on dashboard/progress.
  - `app/_layout.tsx` (Stack screen animation options for transitions).
  - `app/dashboard.tsx`, `app/progress.tsx` (swap/augment with ring).
- **New types/state:** none (presentational).
- **Acceptance criteria:**
  - Correct answer → light success haptic; wrong → warning haptic; no crash on `expo start --web` (guard `Platform.OS !== 'web'`).
  - Progress ring animates on mount; respects light/dark theme colors.
  - Page transitions feel smooth; no dropped-frame jank on a mid-range device in Expo Go.
- **Manual test steps:** Answer correct/wrong in quiz → feel distinct haptics on device. Open dashboard → ring animates from 0 to value. Run on web → no haptics error. Navigate between screens → transitions animate.
- **Git commit:** `feat(ux): add haptics, animated progress ring, and screen transitions`

## 4. Suggested Order of Implementation

```
CP0 → CP-A → CP-B → CP-C → CP-D → CP-E
```

Rationale:
1. **CP0** — safety net first.
2. **CP-A** before everything: shuffle reshapes how the quiz reads questions; do it once, build modes on top.
3. **CP-B** consumes A's `activeQuiz` shape.
4. **CP-C** records results that B now produces (mode/duration).
5. **CP-D** is pure derivation over C's `bestScore`/history.
6. **CP-E** last — polish on a feature-stable base, lowest regression risk, easy to drop if time runs out.

**Two-week cadence (single dev):** Week 1 → CP0 (½d), CP-A (1.5d), CP-B (2d). Week 2 → CP-C (1.5d), CP-D (1.5d), CP-E (1.5d), buffer/QA (1d). If time slips, **CP-E and the bank size are the shed-able scope**; A–D still ship a coherent "sharper evaluation" release.

## 5. Biggest Technical Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| **Shuffle breaks grading** — `correctAnswerIndex` points at original option order. | High | In `startQuiz`, shuffle a *copy* of options and recompute the correct index by tracking the correct option's new position. Add a manual "pick known answer → expect Benar" test in CP-A. |
| **Timer leaks / setState-after-unmount** in Ujian mode (early exit, navigation). | Med | Single `useEffect` interval keyed to `activeQuiz`; clear in cleanup and on finish/exit. Use a ref guard. |
| **`resetDemo` forgets new state** → dirty repeat demos (the cardinal demo sin). | Med | New fields (`activeQuiz`, `attemptHistory`, `bestScore`) added to `initialState`; `resetDemo` already spreads `initialState`, so just keep everything in `initialState`. Add reset to each CP's acceptance test. |
| **Haptics/animation on web** (`expo start --web`) crash or warn. | Med | Guard `Platform.OS !== 'web'` for haptics; reanimated/worklets already configured — verify ring on web renders static if needed. |
| **State shape churn** ripples into `quiz.tsx`/`progress.tsx` reads. | Med | Land CP-A's `activeQuiz` shape early and freeze it; later CPs only append fields. |
| **Typed-routes param drift** (adding `mode` to `/quiz`). | Low | Pass via `router.push({ pathname, params })`; run `tsc` after route param changes (typed routes are on). |
| **Scope creep eating the buffer.** | Med | CP-E + bank size are explicitly the cut line; protect Week-2 QA day. |

## 6. Keeping the App Demo-Ready After Each Checkpoint

A checkpoint is **not done** until all of the following hold — this is the per-CP exit gate:

1. **Green typecheck:** `npx tsc --noEmit` returns zero errors.
2. **Deps still aligned:** `npx expo install --check` says up to date (we add no deps, so this should never regress).
3. **Golden demo flow intact:** the README Demo Flow (Login → … → Reset) runs end-to-end on Expo Go with no red screen.
4. **Reset is clean:** after `resetDemo()`, every screen shows the fresh/empty state — *including the new feature's state*. This is the single most important demo invariant.
5. **Commit on green only:** each CP is one commit; never commit a half-wired state shape. Tag the last green CP (e.g. `git tag v0.2-cpC`) so any demo can roll back to a working build instantly.
6. **Feature is independently showable:** each CP adds something you could demo alone (shuffle, then modes, then history, then readiness, then polish) — so even a partial v0.2 demos well.
7. **Web doesn't crash** (if web is part of the pitch): guard native-only APIs.

> Practical loop per CP: implement → `tsc` → run on device → walk demo flow → hit Reset → confirm clean → commit. Keep `npx expo start -c` running; reload between checks.
